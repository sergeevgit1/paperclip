# Roadmap

## Цель

Сделать систему устойчивее без большого и рискованного рефакторинга:

- сначала остановить повторяющиеся продовые падения
- потом убрать системные источники хрупкости
- только после этого трогать архитектуру глубже

## Принцип

Делаем не идеально, а в порядке максимального снижения риска на единицу усилия.

## Этап 0. Немедленная стабилизация

Срок: сегодня - 1 день

### 1. Запретить создание полусломанных агентов

Что делаем:

- в `agent-hires` валидируем итоговый instructions bundle до возврата `201`
- если bundle не резолвится или не хватает обязательных файлов, создание считается ошибкой

Почему сейчас:

- это самый дешёвый способ прекратить появление новых битых агентов

Ожидаемый эффект:

- не будет новых кейсов типа `NLP/ML Engineer`

### 2. Запретить частичную instructions-конфигурацию

Что делаем:

- если передан хотя бы один instructions-ключ, проверяем целостность набора
- для `managed` режима не допускаем состояние, где mode есть, а root/path нет

Почему сейчас:

- это главный источник ядовитых конфигов в БД

Ожидаемый эффект:

- система перестанет сохранять заведомо неконсистентные записи

### 3. Закрепить корректный preflight только для workspace-bound runs

Что делаем:

- project-level codebase checks применять только к реально workspace-bound runs
- не требовать codebase только из-за наличия `projectId`

Почему сейчас:

- это уже доказанный источник массовых ложных падений

Ожидаемый эффект:

- орг-задачи больше не будут валиться как инженерные

### 4. Добавить admin/audit диагностику

Что делаем:

- endpoint или job, который показывает:
- агентов без валидного bundle
- project-bound issue без usable workspace
- workspace с несуществующим `cwd`
- stale instructions config

Почему сейчас:

- чтобы не ждать очередного падения heartbeat

Ожидаемый эффект:

- сокращение времени диагностики и ручного ремонта

### Критерий завершения этапа

- новый агент не может быть создан в невалидном состоянии
- org issue не падают на `project.codebase_missing`
- есть список уже сломанных сущностей в системе

## Этап 1. Устранение главных источников повторения

Срок: 2-5 дней

### 1. Разделить контекст проекта и исполнительную workspace-привязку

Что делаем:

- вводим явное условие, когда issue требует project workspace
- `projectId` оставляем для контекста
- workspace/codebase requirement делаем отдельным сигналом

Варианты:

- `requiresProjectWorkspace`
- или `project.kind = codebase | operational`

Почему это следующий этап:

- без этого система и дальше будет путать орг- и инженерные задачи

Ожидаемый эффект:

- устранение корневой двусмысленности

### 2. Ослабить автоматическое наследование execution linkage

Что делаем:

- `projectId` можно наследовать
- `projectWorkspaceId` и execution semantics наследуются только явно или по строгому правилу

Почему это важно:

- сейчас один пустой проект заражает дочерние issue каскадом

Ожидаемый эффект:

- прекращение массового распространения неправильной project/workspace linkage

### 3. Добавить явную post-create health validation

Что делаем:

- после create agent, update instructions bundle, create/update project workspace запускать нормализованную валидацию и логировать результат

Почему это важно:

- чтобы ошибки проявлялись в момент изменения конфигурации, а не при первом run

Ожидаемый эффект:

- уменьшение delayed runtime failures

### Критерий завершения этапа

- issue без workspace requirement не попадают под codebase preflight
- child/follow-up issue не получают ложную execution-привязку автоматически
- после create/update система сразу знает, рабочая сущность или нет

## Этап 2. Повышение живучести системы

Срок: 1-2 недели

### 1. Добавить self-healing для managed instructions

Что делаем:

- если bundle есть на диске, а config stale, система автоматически чинит config
- если entry file не совпадает, система восстанавливает корректную ссылку
- если root потерян, но managed bundle найден, система пересобирает bundle state

Почему не в первом этапе:

- сначала нужно ужесточить инварианты, иначе self-healing будет лечить мусорные состояния

Ожидаемый эффект:

- меньше ручных восстановлений bundle

### 2. Добавить self-healing и normalization для managed project workspaces

Что делаем:

- при обнаружении managed workspace со stale path пытаться восстановить корректный runtime path
- для managed project folders использовать вычисляемый canonical path

Почему это полезно:

- устранит кейсы `/home/...` против `/paperclip/...`

Ожидаемый эффект:

- меньше ложных `cwd missing` и `codebase missing`

### 3. Сделать health-check и background audit регулярной операцией

Что делаем:

- периодический job, который помечает подозрительные сущности
- возможно, отдельный admin dashboard

Почему:

- ручной поиск уже не масштабируется

Ожидаемый эффект:

- переход от реактивного к проактивному сопровождению

### Критерий завершения этапа

- часть ошибок repairable без ручного DB-вмешательства
- path drift и stale config выявляются автоматически

## Этап 3. Архитектурное упрощение

Срок: 2-4 недели

### 1. Убрать абсолютный `cwd` как главный источник истины

Что делаем:

- хранить логическую ссылку на managed workspace
- runtime path вычислять из environment-aware resolver
- абсолютный путь делать derived state, а не core state

Почему это нужно:

- абсолютные пути в mixed host/container среде фундаментально хрупкие

Ожидаемый эффект:

- снижение environment-dependent bugs

### 2. Унифицировать схему хранения instructions bundles

Что делаем:

- выбрать один canonical способ: либо только `agent.id`, либо строгое отображение slug -> id без смешения
- мигрировать старые bundle-path conventions

Почему:

- сейчас coexistence `id` и slug-путей делает recovery и portability хрупкими

Ожидаемый эффект:

- проще диагностика, импорт/экспорт и восстановление

### 3. Разделить модель project на разные доменные типы

Что делаем:

- либо разные project kinds
- либо отдельные сущности для operational stream и codebase stream

Почему:

- корень системной двусмысленности в том, что один `project` сейчас выполняет слишком много ролей

Ожидаемый эффект:

- более прозрачное поведение execution pipeline

### Критерий завершения этапа

- модель execution/workspace перестаёт зависеть от случайных metadata-совпадений
- код и runtime semantics становятся предсказуемее

## Что не делать сейчас

### 1. Не делать большой тотальный рефакторинг всего workspace/runtime слоя

Почему:

- высокий риск
- большая цена
- можно получить много регрессий раньше, чем придёт польза

### 2. Не запрещать проекты без кодовой базы

Почему:

- это сломает валидные use cases: onboarding, hiring, planning, operations

### 3. Не убирать inheritance полностью

Почему:

- inheritance полезен
- проблема в смешении контекста и исполнения, а не в наследовании как таковом

## Приоритеты по порядку

1. Fail-fast на создании агента
2. Валидация instructions config
3. Корректный preflight только для workspace-bound runs
4. Audit broken entities
5. Разделение project context и execution requirement
6. Ослабление автоматического execution inheritance
7. Self-healing stale config/path
8. Архитектурная нормализация path/storage model

## Как измерять успех

### 1. Метрики по ошибкам preflight

- падения `instructions.root_missing`
- падения `instructions.missing_agents_md`
- падения `project.primary_workspace_missing`
- падения `project.codebase_missing`

### 2. Метрики качества provisioning

- доля агентов, созданных и успешно прошедших первый run без ручного вмешательства

### 3. Метрики операционной устойчивости

- сколько инцидентов требует ручной правки БД
- сколько требует restart
- среднее время диагностики
